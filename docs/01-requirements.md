# 需求分析

## 项目背景

校园内师生经常发生物品遗失和捡拾场景。传统依赖朋友圈、公告栏和班级群发布信息，存在传播范围有限、状态混乱、身份难核验、认领流程不规范、重复发布较多、管理员无法统一管理等问题。

本系统提供统一的校园失物招领平台，支持失物发布、拾物发布、在线认领、管理员审核、自动过期、重复检测、权限控制和后台统计。

## 用户角色

| 角色 | 权限 |
| --- | --- |
| VISITOR | 浏览公开列表和详情 |
| USER | 注册登录、发布失物 / 拾物、申请认领、查看自己的申请 |
| ADMIN | 审核认领申请、下架物品、查看统计、管理用户 |
| SUPER_ADMIN | 管理管理员和系统配置，作为后续扩展角色 |

## 功能需求

| 模块 | 功能 |
| --- | --- |
| 认证 | 注册、登录、登出、获取当前用户 |
| 用户 | 用户资料、角色、状态管理 |
| 物品信息 | 发布、更新、删除、详情、列表、筛选、分页、排序 |
| 重复检测 | 发布前检测相似标题、描述、地点、类型 |
| 认领申请 | 提交申请、查看申请、取消申请 |
| 审核 | 管理员通过或驳回申请，保证状态一致 |
| 通知 | 认领申请提醒、审核结果通知 |
| 操作日志 | 记录审核、下架、角色变更等关键动作 |
| 定时任务 | 每小时扫描过期物品 |
| 统计 | 物品数量、状态分布、待审核数量 |

## 非功能需求

- 前后端分离，可本地独立启动。
- 接口返回格式统一。
- 表单必须有前后端校验。
- 列表必须支持查询、筛选、分页。
- 后端必须有权限控制和越权防护。
- 核心业务状态流转必须有事务保护。
- 支持课程演示账号和初始化数据。
- 保留 AI 辅助开发说明。

## 核心演示流程

1. 普通用户发布拾物信息。
2. 另一个用户申请认领。
3. 管理员审核通过。
4. 过期物品自动变为 EXPIRED。
5. 相似信息发布时展示重复检测提示。
